# 评估细则

## 评估流程

1. 候选人提交 GitHub 仓库链接（面试前至少 1 天）
2. 面试官提前 review 代码（15 分钟）
3. 候选人 30 分钟现场演示
4. 面试官 15 分钟提问与评分

## 评分细则

### Agent 架构理解（25 分）

| 分数 | 标准 |
|------|------|
| 22-25 | Agent Loop 设计完整（观察-思考-执行循环），有明确的迭代终止条件，错误恢复机制完善（至少 3 轮自动修复），能讲清楚与 OpenHands 原版的差异 |
| 15-21 | 有 Agent Loop 但较为简单（如只做了单轮调用），错误处理基本具备 |
| 8-14 | 只是把用户消息发给 LLM 再执行，没有真正的循环 |
| 0-7 | 不理解 Agent Loop 概念，只做了一次 LLM 调用 |

### 代码质量（20 分）

| 分数 | 标准 |
|------|------|
| 18-20 | 模块化清晰，类型注解完善，错误处理覆盖全面，有单元测试，README 完整 |
| 12-17 | 代码可读，有基本错误处理，目录结构合理 |
| 6-11 | 能跑但代码混乱，大量硬编码，无错误处理 |
| 0-5 | 不可运行或代码完全无组织 |

### LLM 集成能力（15 分）

| 分数 | 标准 |
|------|------|
| 13-15 | Tool Calling 定义合理（schema 设计好），System Prompt 精心设计让 Agent 行为可控，正确管理多轮对话上下文（messages 数组追加 tool results） |
| 8-12 | 能调用 LLM API，有基本 Tool Calling |
| 4-7 | 硬编码回复或 Prompt 设计粗糙 |
| 0-3 | 无法成功调用 LLM |

### 沙箱安全（10 分）

| 分数 | 标准 |
|------|------|
| 9-10 | 命令白名单 + 路径限制 + 超时控制 + 资源限制，至少两重防护 |
| 6-8 | 有基本隔离（子进程 + 临时目录），命令超时 |
| 3-5 | 只有子进程调用，无路径限制 |
| 0-2 | 直接在宿主机执行代码 |

### 前端体验（15 分）

| 分数 | 标准 |
|------|------|
| 13-15 | SSE 流式推送，Agent 思考链可见，工具调用过程可追踪，UI 美观可用 |
| 8-12 | 能流式展示 LLM 输出，基本交互可用 |
| 4-7 | 只有静态展示，无法看到中间过程 |
| 0-3 | 无前端或前端无法正常工作 |

### 演示质量（15 分）

| 分数 | 标准 |
|------|------|
| 13-15 | 三个 demo 流畅跑通（含错误修复场景），架构讲解清晰有深度，能自信回答追问 |
| 8-12 | 跑通 1-2 个 demo，基本讲清架构 |
| 4-7 | Demo 部分成功但讲解不清 |
| 0-3 | Demo 失败，无法解释代码 |

## 总分

| 等级 | 分数 | 建议 |
|------|------|------|
| 优秀 | 85-100 | 强烈推荐，可定级高级 |
| 良好 | 70-84 | 推荐录用，中级 |
| 及格 | 55-69 | 可考虑，初级/中级 |
| 不及格 | 0-54 | 不推荐 |
